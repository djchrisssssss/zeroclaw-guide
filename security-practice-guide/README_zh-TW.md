專為 **高許可權自主智慧體 (OpenClaw)** 量身定製的權威安全實踐指南。它將傳統“主機靜態防禦”的正規化轉變為“智慧體零信任架構 (Zero-Trust Architecture)”，有效應對破壞性操作、提示詞注入、供應鏈投毒和高危業務邏輯執行等智慧體專屬風險。

## 🎯 適用場景與核心原則

> **本指南是面向 OpenClaw 本身（Agent-facing）的，不是傳統“僅供人類手動操作”的加固清單。**  
> 實際使用中，你可以把本指南直接發給 OpenClaw，讓它先評估可靠性，再自動完成防禦矩陣部署，大幅降低手工配置成本。

> **重要邊界說明：本指南並不能讓 OpenClaw“完全安全”。**  
> 安全是複雜的系統工程，不存在絕對安全。  
> 本指南只在其定義的威脅模型、適用場景和操作假設下發揮作用。  
> **最終的安全兜底與關鍵判斷，仍然在使用者自己。**

### 適用場景
- OpenClaw 執行在高許可權環境（具備終端 / Root 能力）
- OpenClaw 持續安裝並使用 Skills / MCPs / 指令碼 / 工具
- 目標是在能力最大化前提下，實現風險可控與審計可追溯

### 核心原則
1. **日常零摩擦**：儘量降低使用者手工安全配置負擔，且使用者日常操作幾乎無感，除非遇到指南認為的紅線  
2. **高危必確認**：不可逆或敏感操作必須暫停並由人類確認  
3. **每晚顯性化巡檢**：所有核心指標都要明確彙報（包含綠燈項）  
4. **預設零信任**：始終假設提示詞注入、供應鏈投毒、業務邏輯濫用可能發生  

### 模型建議（重要）
這份指南主要由 OpenClaw 理解並執行。  
建議優先使用**最新、推理能力更強的模型**（例如 Gemini / Opus / Kimi / MiniMax 等系列中的高階模型）。  
更“聰明”的模型通常在以下方面明顯更穩：
- 長上下文安全約束理解
- 隱性指令/注入模式識別
- 部署步驟一致性與低錯誤率執行

✅ 這也是本指南能**顯著降低使用者配置成本**的關鍵：OpenClaw 可自行完成大部分“理解 → 部署 → 驗證”流程。

## 🌟 為什麼需要本指南？

讓類似 OpenClaw 這樣的 AI Agent 擁有 Root 或終端訪問許可權極其強大，但也伴隨著巨大的固有風險。傳統的安全措施（如 `chattr +i`、防火牆）要麼與 Agent 的自動化工作流不相容，要麼不足以抵禦針對大語言模型 (LLM) 的特有攻擊（如 Prompt Injection）。

本指南提供了一套經過實戰檢驗的、極簡的 **三層防禦矩陣**：
1. **事前 (Pre-action)**: 行為黑名單與嚴格的技能包安裝審計協議（防供應鏈投毒）
2. **事中 (In-action)**: 許可權收窄與跨技能業務風控前置檢查 (Pre-flight Checks)
3. **事後 (Post-action)**: 每晚自動化顯性巡檢（覆蓋 13 項核心指標）與大腦 Git 災備同步

## 🚀 極簡部署流程 (Zero-Friction Flow)

在 AI 時代，部署安全防線不應該由人類手動敲程式碼。**您可以讓您的 OpenClaw 為您自動完成一切：**

1. **下載指南**: 下載核心文件 [OpenClaw 極簡安全實踐指南.md](docs/OpenClaw極簡安全實踐指南.md)
2. **傳送給 Agent**: 在聊天視窗中，將該 markdown 檔案直接傳送給您的 OpenClaw Agent
3. **讓 AI 審計**: 向您的 Agent 傳送指令：“*請仔細閱讀這份安全指南，評估它是否可靠？*”
4. **一鍵部署**: 在 Agent 確認指南可靠後，傳送指令：“*請完全按照這份指南，為我部署防禦矩陣。包括寫入紅/黃線規則、收窄許可權，並部署夜間巡檢 Cron Job。*”
5. **驗收測試（可選）**: 部署完成後，請按照 [驗證與攻防演練手冊](docs/Validation-Guide-zh-TW.md) 對 Agent 進行一次突擊測試，確保紅線生效

*(注：本倉庫 `scripts/` 目錄下的巡檢指令碼僅作開源展示與人類參考之用，**您完全不需要手動複製或執行它**，Agent 會自動從指南文件中提取邏輯併為您編寫部署。)*

## ⚠️ 免責宣告與常見問題（Disclaimer & FAQ）

### 1. 適用範圍與能力前提

本指南假定執行者（人類或 AI Agent，尤其是 AI Agent）具備以下能力：

- 理解 Linux 基礎系統管理概念（檔案許可權、chattr、cron 等）
- 能夠準確區分紅線/黃線/安全命令的邊界
- 能夠在執行前理解命令的完整語義和副作用

**如果執行者（尤其是 AI 模型）不具備上述能力，請勿直接套用本指南。** 能力不足的模型可能誤解指令含義，導致比不配置安全策略更嚴重的後果。

### 2. AI 模型執行風險

本指南的核心機制"行為層自檢"依賴 AI Agent 自主判斷命令是否觸及紅線，這帶來以下固有風險：

- **誤判**：能力較弱的模型可能將安全命令誤判為紅線（阻斷正常工作），或將危險命令誤判為安全（導致安全事故）
- **理解偏差**：模型可能對紅線命令做過於字面的匹配（只攔 `rm -rf /` 但不攔 `find / -delete`），或過於寬泛（把所有 `curl` 都當紅線）
- **執行偏差**：模型在實施 `chattr +i` 等防護措施時，錯誤的引數可能導致系統不可用（如鎖錯檔案影響 OpenClaw 正常執行）
- **指南被注入**：如果本指南以 prompt 形式注入 Agent，惡意 Skill 可能透過 prompt injection 篡改指南內容，使 Agent "認為"紅線規則已被修改

**本指南作者不對因 AI 模型誤解、誤執行本指南內容所造成的任何損失承擔責任，包括但不限於：資料丟失、服務中斷、配置損壞、安全漏洞暴露、金鑰洩露。**

### 3. 非銀彈宣告

本指南提供的是**縱深防禦的基礎框架**，不是完整的安全解決方案：

- 不能防禦 OpenClaw 引擎本身、底層 OS 或依賴元件的未知漏洞
- 不能替代專業安全審計（生產環境或涉及真實資產的場景應另行評估）
- 巡檢屬於事後檢測，只能發現已發生的異常，無法回滾已造成的損害

### 4. 環境假設

本指南基於以下環境編寫，偏離時請自行評估：

- 單使用者、個人用途的 Linux 伺服器
- OpenClaw 以 root 許可權執行，追求能力最大化
- 網路可達 GitHub（Git 備份）和 Telegram 等 API（巡檢通知）

### 5. 版本與時效性

本指南基於編寫時的 OpenClaw 版本，後續版本可能引入原生安全機制使部分措施過時或衝突。請定期檢查相容性。

## 常見問題（FAQ）

### 💡 實驗與體驗篇

#### Q1: 這份指南是一次怎樣的實驗？為什麼不直接做成 Skill？
**這是一次給 AI 植入安全“思想鋼印”的實驗。**
我們嘗試過構建專門的安全 Skill，但發現直接向 OpenClaw 腦中植入包含“事前、事中、事後”策略的 Markdown 鋼印更有意思。Skill 本質上只是掛載的工具，而“思想鋼印”能夠改變 Agent 的基礎認知。如果你確實需要一個 Skill，你也完全可以透過多與它對話讓它自己生成一個。總之，只要機器不金貴，盡情折騰。

#### Q2: 部署後 OpenClaw 會變得束手束腳嗎？
**取決於你與模型的磨合，請務必找到平衡點（強烈建議不要被束手束腳，會煩死）。**
特別像 OpenAI 系的模型本身就很嚴格。如果你完全順著它的思路走，它可能什麼都不敢幹。安全和業務永遠是權衡：安全太過不好，沒有也不好。這就是我們在“核心原則”中強調“日常零摩擦”的原因。如果覺得防線過緊，由於模型差異，你在部署前可以多和你的 🦞 對話，把擔憂和需求溝通清楚，再讓其落地。

#### Q3: 這份指南只針對 Linux Root，我的環境是 Mac / Win 怎麼辦？
**沒有完美適配，但有 Trick（投餵技巧）。**
你可以把 `OpenClaw極簡安全實踐指南.md` 直接餵給你的 OpenClaw，因為大模型擁有舉一反三的能力。模型會自動給你關於系統相容性的建議，然後**你可以直接讓它嘗試為你生成一份“適配後”的專屬指南**，再考慮是否要落地。

#### Q4: 植入安全“思想鋼印”後，還有什麼進階樂趣？
一旦你的 Agent 理解了這份指南背後的安全設計理念，此後如果你給它安裝其他優秀的安全 Skill 或企業級解決方案，更有意思的化學反應就會發生：**你的 OpenClaw 會主動根據它腦海裡的這層安全記憶，去對比、打分並分析那些新來的安全工具。**

#### Q5: 災備（Git Backup）部分是強制的嗎？
**不是必選。**
災備的必要性取決於你個人對記憶與隱私資料的在意程度。如果你只希望保障執行時安全，不希望遠端同步資料，完全可以直接刪掉那個機制。你甚至可以讓 Agent 先對敏感資訊“加密後再備份”。

### 🔧 實操與技術篇

#### Q6: 我用的模型比較弱（如小引數模型），能用這個指南嗎？
**不建議直接使用完整指南。** 行為層自檢要求模型能準確解析命令語義、理解間接危害、在多步操作中保持安全上下文。如果模型做不到，建議：只使用 `chattr +i`（純系統級，不依賴模型能力），並將 Skill 安裝安檢交由人類手動完成。

#### Q7: 紅線列表是否完備？
**不可能完備。** Linux 下實現同一破壞效果的方式很多（`find / -delete`、Python 指令碼刪除、DNS 隧道外發資料等）。指南中"拿不準按紅線處理"是兜底原則，但最終依賴模型的判斷能力。

#### Q8: Skill 安檢是否只需要做一次？
不是。Skill 更新、OpenClaw 引擎更新、Skill 行為異常、巡檢指紋校驗不匹配時都需要重新安檢。

#### Q9: `chattr +i` 會不會影響 OpenClaw 正常執行？
**可能會。** `openclaw.json` 加鎖後 OpenClaw 自身也無法更新該檔案，升級或配置變更會報 `Operation not permitted`。需要修改時先 `sudo chattr -i` 解鎖，改完再重新加鎖。另外，**絕對不要對 `exec-approvals.json` 加鎖**（指南中已說明），否則引擎執行時寫入後設資料會失敗。

#### Q10: 如果模型誤執行了 `chattr +i` 到錯誤的檔案怎麼辦？
手動修復：
```bash
# 查詢所有被設定了 immutable 屬性的檔案
sudo lsattr -R /home/ 2>/dev/null | grep '\-i\-'

# 解鎖誤鎖的檔案
sudo chattr -i <檔案路徑>
```
如果誤鎖了關鍵系統檔案（如 `/etc/passwd`），可能需要進入 recovery mode 修復。

#### Q11: 巡檢指令碼本身會不會有安全風險？
巡檢指令碼以 root 許可權執行，如果被篡改就等於一個每晚自動執行的後門。建議對巡檢指令碼本身也做 `chattr +i` 保護，Telegram Bot Token 單獨存放並設為 `chmod 600`。

#### Q12: 如果 OpenClaw 引擎本身有安全漏洞怎麼辦？
本指南的防護措施都建立在"引擎本身可信"的假設上，無法防禦引擎層漏洞。建議關注 OpenClaw 官方安全公告，及時更新。
